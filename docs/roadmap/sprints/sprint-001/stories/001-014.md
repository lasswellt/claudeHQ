# 001-014: Hub SQLite database with migrations

## Context
The Hub persists all state in SQLite via better-sqlite3. This story creates the database connection, migration system, and initial schema (machines, sessions tables).

## Requirements
1. Create `packages/hub/src/db.ts` — database connection + migration runner
2. Enable WAL mode and recommended PRAGMAs on connection
3. Create migration system: numbered SQL files in `packages/hub/src/migrations/`
4. Migration 001: `machines` table (id, display_name, last_seen, status, max_sessions, meta)
5. Migration 002: `sessions` table (id, machine_id, prompt, cwd, flags, status, pid, exit_code, claude_session_id, parent_session_id, started_at, ended_at, last_activity_at, recording_path, recording_size_bytes, recording_chunk_count, created_at) + indexes
6. Migration 003: `queue` table + `notifications` table + `notification_config` table
7. Export typed query functions (DAL): `insertSession`, `updateSession`, `getSession`, `listSessions`, etc.

## Acceptance Criteria
- [ ] Database created at configured path on first run
- [ ] WAL mode enabled: `PRAGMA journal_mode` returns 'wal'
- [ ] All PRAGMAs set: synchronous=NORMAL, cache_size=-20000, busy_timeout=5000, foreign_keys=ON
- [ ] Migrations run in order, tracked in a `_migrations` table
- [ ] Re-running migrations is idempotent (skip already applied)
- [ ] All queries use parameterized prepared statements
- [ ] `db.close()` called on graceful shutdown

## Technical Notes
- `const db = new Database(path)` — better-sqlite3 synchronous API
- PRAGMAs: run immediately after opening connection
- Migration files: `001_machines.sql`, `002_sessions.sql`, `003_queues_notifications.sql`
- Track applied migrations: `CREATE TABLE IF NOT EXISTS _migrations (id INTEGER PRIMARY KEY, name TEXT, applied_at INTEGER)`
- INTEGER for timestamps (unixepoch()), TEXT for UUIDs
- See architecture doc for full schema definitions
