# 001-007: Shared record types

## Context
These are the core data types that represent entities in the system. Every package imports these. They must be defined first because WebSocket messages and API routes reference them.

## Requirements
1. Define `SessionRecord` with Zod schema: id, machine_id, prompt, cwd, flags, status, pid, exit_code, claude_session_id, parent_session_id, started_at, ended_at, last_activity_at, recording_path, created_at
2. Define `SessionStatus` enum: `queued | running | completed | failed`
3. Define `MachineRecord`: id, display_name, last_seen, status, max_sessions, meta (version, os, arch)
4. Define `MachineStatus` enum: `online | offline`
5. Define `QueueTask`: id, machine_id, prompt, cwd, flags, priority, position, created_at
6. Define `NotificationRecord`: id, session_id, type, channel, payload, sent_at, delivered
7. Define `NotificationConfig`: id, webhooks, events, enabled
8. Export all types and schemas from `@chq/shared`

## Acceptance Criteria
- [ ] All Zod schemas compile and produce correct TypeScript types via `z.infer<>`
- [ ] `import { SessionRecord, MachineRecord } from '@chq/shared'` works in agent and hub
- [ ] Each schema validates correct data and rejects invalid data (test)
- [ ] Timestamps are `z.number()` (Unix epoch seconds)
- [ ] IDs are `z.string()` (UUIDs)

## Technical Notes
- Use `z.object({...})` for each record type
- Export both the schema (`sessionRecordSchema`) and inferred type (`SessionRecord`)
- Use `z.enum()` for status types
- `meta` on MachineRecord: `z.object({ version: z.string(), os: z.string(), arch: z.string() }).optional()`
- `flags` on SessionRecord/QueueTask: `z.array(z.string()).optional()`
