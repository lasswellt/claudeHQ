# 001-012: PTY pool — concurrent session management

## Context
The Agent manages multiple concurrent PTY sessions up to a configured limit. The pool tracks active sessions, enforces capacity, and provides lookup by session ID.

## Requirements
1. `PtyPool` class with:
   - `maxSessions` config (from AgentConfig)
   - `spawn(opts)` → creates PtySession, adds to pool, returns session ID
   - `get(sessionId)` → returns PtySession or undefined
   - `kill(sessionId)` → kills specific session
   - `killAll()` → kills all sessions (for graceful shutdown)
   - `list()` → returns array of active session summaries
   - `activeCount` → number of running sessions
   - `hasCapacity` → boolean (activeCount < maxSessions)
2. Events: `session:started`, `session:output`, `session:ended` (for WS relay)
3. Auto-remove completed sessions from active pool after exit

## Acceptance Criteria
- [ ] Pool enforces maxSessions limit (spawn returns error if full)
- [ ] Multiple sessions run concurrently without interference
- [ ] `pool.kill(id)` terminates the correct session
- [ ] `pool.killAll()` waits for all sessions to terminate
- [ ] Session events propagate to pool event emitter
- [ ] Completed sessions cleaned up from pool

## Technical Notes
- Use a `Map<string, PtySession>` for session storage
- Extend `EventEmitter` or use a typed event pattern
- Session IDs: generate with `crypto.randomUUID()`
- On session end: emit event, remove from map, check queue for auto-advance (later epic)
